Base Protocol

Carriers

This protocol is carrier independent. This does not rely on Internet Protocol and can be used in other communication systems. As long as the data remains the same, information can be forwarded between hosts. Each client and server will have a short semi-unique identification number. Since this protocol will allow forking and multiple players per client, each player will have an identification. If one client hosts 4 players, they will all share the same address. However, if the server wants to send a message to a certain player then it will add the client's identification number. The message will be sent to the client which will then route the data correctly to the other players and anyone who is forked.

Sizes

The protocol allows for byte order changes and is dynamic depending on the circumstances. Each packet transfered may contain multiple packets together or they can be separated. Before this goes into details, there are a specific number of bytes each field has. The following are valid sizes that can be used.

  • Signed 8-bit Integer (-128 to 127)
  • Unsigned 8-bit Integer (0 to 255)
  • Signed 16-bit Integer (-32,768 to 32,767)
  • Unsigned 16-bit Integer (0 to 65,535)
  • Signed 32-bit Integer (-2,147,483,648 to 2,147,483,647)
  • Unsigned 32-bit Integer (0 to 4,294,967,295)
  • Signed 64-bit Integer (-9,223,372,036,854,775,808 to 9,223,372,036,854,775,807)
  • Unsigned 64-bit Integer (0 to 18,446,744,073,709,551,615)

Definition

The base packet is dynamic in size and both fields determine the byte order and the actual size of the base packet. The first two fields always exist:
Offset
Size
Description
0 4 Packet Identification:
This is an ASCII string of 4 characters excluding the NULL terminator character. It may only consist of letters from 'A' to 'Z' and 'a' to 'z'. The first character must always be lowercase while the remaining characters are upper case. If the first character is lower-case then the byte order matches the receiving end, other if the third character is lowercase then the receiving end uses a reversed byte order. This makes it so clients and servers on the same byte order do not have to translate anything even if they use the same byte order. This protocol as stated before is byte order independent. An example of a matching byte order identification would be "cHAT" where if a system with a reversed byte order received it, then it would be "ATcH".
4 4 Flags
These flags determine whether fields in the packet header exist or not, they may also determine how the data is laid out. The byte order of the flag field matches the byte order of the packet identification string. They are as followed:
  • Compression (0x00000007): The data field is compressed to save more space.
    • No Compression (0x00000000)
    • RLE Compression (0x00000001)
    • GZip Compression (0x00000002)
    • BZip Compression (0x00000003)
    • LZMA Compression (0x00000004)
    • Huffman Compression (0x00000005)
  • Encryption (0x00000038): The data field is encrypted to protect data. Note that only the data is encrypted so anybody can guess what data packets contain to get the encryption key. There is no indication to verify whether unencrypted data is correct or not, it's valid if the contained data is valid.
    • No Encryption (0x00000000)
    • Exclusive-OR Encryption (0x00000008)
    • AES 128-bit Encryption (0x00000010)
    • AES 192-bit Encryption (0x00000018)
    • AES 256-bit Encryption (0x00000020)
  • Extended Packet Size (0x00000040): Instead of using UInt16s to represent the size of the data in a packet, UInt32s are used.
  • Marker (0x0000800): Void all other flags, each packet only consists of 8 bytes.
  • Host Identity (0x00000100): This flag contains a host identity for multi-player clients to identify who this packet is really for.
  • Error Checking (0x00000200): The data in the packet contains error checking, after every 8 bytes there is another byte indicated even/odd numbers of the previous data. If compression is used, error checking is added to the compressed packet only and the compressed size is increased, otherwise if no compression is used then the size is increased. ErrorByte = ((Data[0] & 1) << 7) | ((Data[1] & 1) << 6) | ((Data[2] & 1) << 5) | ((Data[3] & 1) << 4) | ((Data[4] & 1) << 5) | ((Data[5] & 1) << 3) | ((Data[6] & 1) << 2) | ((Data[7] & 1) << 1).
  • Not Important (0x00000400): The packet is not important and can be dropped. If a client or server is under load, packets with this flag may be ignored to reduce load.
  • Padding (0x00000800): This packet is padded to a 4-byte boundary. If the packet contains data, the size of the data is unaffected even when padding is placed in.
  • Force Reverse Byte Order (0x00001000): The byte order of the packet is swapped, note that the flags field remains the same, the following fields are swapped.
  • Don't Queue For Client/Server Synchronization (0x00002000): Indicates that the packet should not be synchronized by the client or server. When synchronization and prediction is enabled in a network game, this packet takes effect and due to this flag. Please note that if you modify ReMooD to add this to every packets such as firing of a weapon and synchronization and prediction is enabled by the server, your shots will most likely miss unless the object is not moving. This action may also cause the network status to be destabilized. This flag does not apply for all types of packets.
  • Force Packet Ordering (0x00004000): This forces the packet to be ordered, if the order number is lower than the number of packets received then the packet is out of order. If there have been dropped packets since the last packet number synchronization then ordering may fail.
  • Try Not to Respond (0x00008000): It is not important for the other side to respond back to this request. This is similar to the Not Important flag but is determined by the other side, if it's network load is high it might not respond back.
  • Contains a Synchronization Time (0x00010000): The packet contains a number indicating the Game Tic to when this packet should be handled. This is mostly used for synchronization and prediction code.
  • Resynchronize Packet Order (0x00020000): This packet contains a number to resynchronize the packet order number with the other side. Since packets might be dropped this is important for packet ordering.
  • Echo (0x80000000): The flags of this packet should not be taken literally, this is for determining what a host supports packet wise. If this flag is set then Marker is an implied flag.
Depending on the setting of certain flags, the remaining packet is dynamic in size. If the Marker flag is set then the following does not exist. Otherwise, it is as followed:
Offset Range

This is the packet number of the flag to determine if it in order or not.
8 to 12 4 Resynchronize Packet Count: (Resynchronize Packet Order)
This identifies the number of this packet so the other side can synchronize it for this client.

Data then follows the packet header if the size is greater than zero and Marker is not set. Depending on which flags are set, the size of the header may range from 8 bytes to 30 bytes.

Support

If a packet contains an unsupported compression method and/or encryption method then the packet is dropped.
